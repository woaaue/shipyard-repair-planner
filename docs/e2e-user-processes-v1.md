# E2E-процессы пользователей (v1)

## 1. Цель документа
Зафиксировать сквозные пользовательские процессы (end-to-end) по всей системе: кто начинает процесс, кто подхватывает, чем процесс завершается, какие API и роли участвуют.

## 2. Роли в контуре
- CLIENT: подача заявки, контроль хода, приемка результата.
- WORKER: выполнение назначенных работ.
- MASTER: постановка задач, приемка/возврат работ, эскалации.
- OPERATOR: управление ремонтом внутри дока.
- DISPATCHER: распределение нагрузки по докам/операторам.
- ADMIN: доступы, орг-структура, аудит, инциденты.

## 3. E2E-01: Заявка клиента -> решение диспетчера
1. CLIENT создает заявку на ремонт.
2. Система валидирует обязательные поля и сохраняет заявку со статусом `SUBMITTED`.
3. DISPATCHER видит заявку в своей очереди.
4. DISPATCHER принимает решение:
5. Вариант A: одобряет, назначает док/оператора (или только док, если так принято в процессе).
6. Вариант B: отклоняет с причиной.
7. CLIENT получает обновленный статус и комментарий.

Текущий API-контур:
- `POST /api/repair-requests`
- `GET /api/repair-requests`
- `PATCH/POST статус заявки` (через существующий endpoint смены статуса)

## 4. E2E-02: Одобренная заявка -> запуск ремонта
1. При одобрении заявки создается сущность ремонта.
2. Ремонт связывается с заявкой и доком.
3. Назначается OPERATOR (если в процессе оператор назначается на этапе диспетчера/сразу после).
4. Ремонт получает стартовый статус (например, `PLANNED`/`IN_PROGRESS` по вашим правилам).
5. CLIENT видит, что заявка перешла в фазу ремонта.

Текущий API-контур:
- `GET /api/repairs`
- `PATCH /api/repairs/{id}/operator`
- `PATCH /api/repairs/{id}/status`

## 5. E2E-03: Планирование работ мастером
1. OPERATOR определяет мастера(ов) в рамках своего дока.
2. MASTER создает/структурирует `WorkItem` по ремонту.
3. MASTER назначает исполнителей (WORKER) через `assignee`.
4. WORKER получает задачи в личной очереди.

Текущий API-контур:
- `GET /api/work-items?repairId=...`
- `POST /api/work-items`
- `PATCH /api/work-items/{id}/assignee`
- `GET /api/work-items?assigneeId=...`

## 6. E2E-04: Выполнение задач рабочим -> ревью мастером
1. WORKER переводит задачу в работу (`IN_PROGRESS`).
2. После выполнения WORKER ставит `COMPLETED`.
3. Система автоматически переводит `reviewStatus` в `PENDING_REVIEW` (если не было `APPROVED`).
4. MASTER проверяет результат:
5. Вариант A: `APPROVED` -> задача фиксируется как завершенная.
6. Вариант B: `REJECTED` -> задача возвращается в `IN_PROGRESS`.
7. Прогресс ремонта пересчитывается от фактических задач.

Текущий API-контур:
- `PATCH /api/work-items/{id}/status`
- `PATCH /api/work-items/{id}/review`

## 7. E2E-05: Исключения (простой/дефект/риск)
1. WORKER или MASTER фиксирует проблему в ходе работ.
2. MASTER классифицирует и дает первичную оценку влияния.
3. OPERATOR принимает локальное решение.
4. Если риск междоковый/системный, OPERATOR эскалирует DISPATCHER.
5. DISPATCHER принимает верхнеуровневое решение (приоритет, перенос, ресурсы).
6. CLIENT получает корректное обновление по срокам/статусу.

Примечание:
- Этот контур частично реализован, но требует унифицированной модели эскалаций и журналирования решений.

## 8. E2E-06: Завершение ремонта -> приемка клиентом
1. Все критичные `WorkItem` завершены и приняты мастером.
2. OPERATOR переводит ремонт в финальный статус.
3. CLIENT выполняет явное действие приемки результата.
4. Система закрывает цикл заявки/ремонта.

Текущий API-контур:
- `POST /api/repair-requests/{id}/accept` (клиентская приемка)

## 9. E2E-07: Отчетность и экспорт
1. Пользователь выбирает тип отчета, период и scope (по роли).
2. Система строит выборку по зоне ответственности:
3. OPERATOR: только свои доки/ремонты.
4. DISPATCHER: несколько доков/операторов.
5. ADMIN: весь контур.
6. Отчет формируется в файл (xlsx/csv) и скачивается.

Текущий API-контур:
- `GET /api/reports/export?...`
- Ролевые ограничения и фильтры по scope.

## 10. E2E-08: Администрирование доступа и структуры
1. ADMIN создает/блокирует/разблокирует пользователей.
2. ADMIN назначает подчиненность `reportsTo` по цепочке:
3. `worker -> master -> operator -> dispatcher`.
4. ADMIN назначает доки ролям, которым это требуется.
5. ADMIN делает reset пароля по обращению.

Текущий API-контур:
- `GET/POST/PATCH /api/users...`
- `POST /api/users/{id}/block`
- `POST /api/users/{id}/unblock`
- `POST /api/users/{id}/reset-password`
- `GET /api/users/{id}/subordinates`

## 11. E2E-09: Админ-контур контроля (качество + инциденты)
1. ADMIN открывает «Качество данных» и видит:
2. Пользователей без руководителя.
3. Пользователей без дока.
4. Нарушения орг-цепочки.
5. Дубли email.
6. ADMIN исправляет проблемы прямо из экрана.
7. ADMIN открывает «Инциденты» и разбирает рисковые события из аудита.

Текущий UI/API-контур:
- `GET /api/admin/data-quality`
- `GET /api/audit-logs`
- `PATCH /api/users/{id}` для исправлений структуры.

## 12. Границы ответственности (чтобы не размывать роли)
1. CLIENT не управляет внутренними задачами и назначениями.
2. WORKER не назначает исполнителей и не меняет орг-структуру.
3. MASTER не распределяет доки и операторов между доками.
4. OPERATOR не решает междоковое распределение ресурсов.
5. DISPATCHER не подменяет мастера в операционном ревью задач.
6. ADMIN не управляет производственным процессом ремонта вместо диспетчера/оператора.

## 13. Что еще блокирует полностью «чистый» e2e
1. Формализованный единый контур эскалаций (тип, уровень, SLA, решение).
2. Более явный lifecycle заявок с обязательной причиной отклонения.
3. Единый журнал решений по ремонту (кто/что/почему) в удобном UI.
4. Полная синхронизация всех экранов прогресса ремонта с вычислением от `WorkItem`.

## 14. Рекомендуемый порядок дальнейшей доработки
1. Довести заявку до полностью управляемого диспетчерского triage (одобрение/отклонение/причина).
2. Завершить контур создания и назначения задач мастером без обходных сценариев.
3. Закрыть e2e исключений: простои/дефекты -> эскалации -> решение.
4. Укрепить клиентский финал: явная приемка, прозрачный итоговый отчет.
5. Дожать админ-контур: инциденты, аудит, quality-fixes как единый operational tool.
