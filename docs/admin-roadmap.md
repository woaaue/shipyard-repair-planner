# Админ в Shipyard Repair Planner: цель, границы и план

## 1. Контекст и итог договоренности
Этот документ фиксирует нашу рабочую договоренность по роли администратора, чтобы при любом перезапуске контекста цель не терялась.

Ключевой вывод:
- В этой системе базовая бизнес-модель уже задана в ядре (иерархия ролей и основные процессы).
- Поэтому админ не должен быть "конструктором бизнес-процесса".
- Роль админа = доступы, орг-структура, контроль, аудит, поддержка.

## 2. Зачем роль админа вообще нужна
Админ нужен не для ежедневной доковой операционки, а для эксплуатации и надежности системы:
1. Управление учетными записями и доступом.
2. Привязка пользователей к орг-структуре и докам по фиксированным правилам.
3. Контроль корректности данных и прав.
4. Разбор инцидентов и пользовательская поддержка.
5. Аудит изменений и действий.

## 3. Что админ не должен делать
1. Не менять базовую иерархию ролей:
- worker -> master -> operator -> dispatcher.
2. Не редактировать произвольно бизнес-workflow ремонта и приемки.
3. Не подменять диспетчера/оператора/мастера в операционных решениях по ремонту.

## 4. Текущий функционал админа (как есть сейчас)
1. Доступ к разделам:
- Пользователи, Настройки, Ремонты, Суда, Календарь, Отчеты.
2. Пользователи:
- список, карточка, блок/разблок, reset пароля, назначение руководителя.
3. Настройки:
- вкладки "Общие", "Доки", "Типы ремонтов".
- "Доки" частично работает как реальная настройка через API.
- "Типы ремонтов" сейчас скорее UI-слой, чем сильная админ-функция.
4. Ограничения доступа:
- есть ролевые ограничения по роутам/части API.

## 5. Главная проблема текущей админки
Админ-экран местами выглядит как "конфигуратор ради конфигуратора" и не закрывает ключевые эксплуатационные сценарии:
- нет фокуса на инцидентах,
- нет наглядного контроля качества данных,
- аудит недостаточно оформлен как рабочий инструмент админа,
- часть настроек малоценна для заданной фиксированной модели.

## 6. Целевая модель админа
### 6.1 Основные зоны ответственности
1. Пользователи и доступ:
- create/activate/block/unblock/reset.
2. Орг-структура:
- назначение руководителя по жесткой цепочке,
- привязка к доку для нужных ролей.
3. Контроль прав:
- проверка соответствия роли и фактического доступа.
4. Аудит:
- кто/что/когда/где/было->стало.
5. Инциденты:
- быстрый разбор проблем доступа и спорных изменений.
6. Качество данных:
- поиск и исправление структурных аномалий.
7. Поддержка пользователей:
- быстрые helpdesk-действия.

### 6.2 Минимально полезное логирование для админа
Для событий в аудите:
1. Кто: userId, email, role.
2. Что: тип действия (LOGIN_FAIL, USER_BLOCK, STATUS_CHANGE, ASSIGN_SUPERVISOR и т.д.).
3. Где: entity + entityId.
4. Когда: timestamp + timezone.
5. Было/стало: oldValue/newValue (для изменений).
6. Контекст: IP, user-agent, request-id.

## 7. Матрица "что есть / что надо"
| Область | Что есть сейчас | Что надо |
|---|---|---|
| Пользователи | список, карточка, блок/разблок, reset | массовые фильтры, быстрые сценарии поддержки |
| Орг-структура | назначение руководителя в карточке | отдельный обзорный экран + строгая валидация цепочки |
| Привязка к доку | частично через формы | единая точка управления привязками |
| Контроль прав | ограничения роутов/API | экран "эффективные права" и проверка конфликтов |
| Настройки | общие + доки + типы ремонтов | очистка от слабополезных настроек |
| Аудит | основа есть, UX слабый | полноценный журнал с фильтрами и деталями |
| Инциденты | вручную через разные экраны | легкий "центр инцидентов" |
| Качество данных | нет выделенного модуля | отчеты-валидаторы аномалий |

## 8. Поэтапный план реализации
### P1 (обязательно, ближайшие PR)
1. Переформатировать админ-раздел вокруг реальных задач эксплуатации.
2. Усилить блок пользователей:
- фильтры проблемных профилей,
- быстрые действия block/unblock/reset.
3. Добавить валидацию орг-цепочки при назначениях.
4. Сделать MVP аудита:
- таблица событий,
- фильтры,
- деталка события.

### P2 (следующая волна)
1. Экран контроля качества данных:
- без руководителя,
- без дока,
- неверная иерархическая связка,
- дубли.
2. Экспорт аудита и отчетов качества.
3. Улучшенный обзор орг-структуры (быстрые назначения из таблицы).

### P3 (дальше, по готовности backend)
1. Легкий центр инцидентов (карточки разбора).
2. Расширенный аудит с корреляцией событий между сущностями.
3. Дополнительные служебные инструменты поддержки.

## 9. Что делать в первую очередь в коде
Следующий практический шаг в фронтенде:
1. Привести страницу админа к 3 блокам:
- Пользователи,
- Орг-структура,
- Аудит.
2. Убрать/свернуть второстепенные настройки, не дающие операционной пользы.
3. Добавить виджет "Проблемы данных" с быстрыми переходами на исправление.

## 10. Критерии готовности админ-модуля (Definition of Done)
1. Админ решает 80% типовых обращений без участия разработчика.
2. Любое спорное изменение можно отследить в аудите.
3. Невалидные орг-связи не сохраняются.
4. Есть быстрый список проблем качества данных.
5. UI админа содержит только функционал, соответствующий его роли в фиксированной модели.

## 12. Управление Верфями И Доками (Новый Контур)
Этот блок фиксирует новые правила для инфраструктурной части админки.

### 12.1 Базовые правила (Шаг 1)
1. Админ может создавать верфь.
2. Админ может редактировать верфь (название, адрес, статус).
3. Админ может только деактивировать/активировать верфь.
4. Физическое удаление верфи запрещено.
5. Название верфи уникально.

### 12.2 Ограничения целостности
1. Нельзя деактивировать верфь, если к ней привязаны активные доки.
2. Нельзя деактивировать верфь, если в ее доках есть активные ремонты.
3. Нельзя деактивировать док, если в нем есть активные ремонты.
4. Для каждой блокировки возвращается понятная причина в API-ответе.

### 12.3 Поэтапная реализация
1. Шаг 1 (текущий): зафиксировать правила и ограничения в документации.
2. Шаг 2: backend-валидация деактивации доков и верфей.
3. Шаг 3: UI-изменения для верфей (без удаления, только activate/deactivate).
4. Шаг 4: аудит действий по верфям/докам (кто, когда, что изменил).
