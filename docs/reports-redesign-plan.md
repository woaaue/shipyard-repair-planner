# Переработка системы отчетов: план внедрения

## 1. Цель
Собрать единый модуль отчетов, который:
- формируется из реальных backend-данных;
- учитывает роль пользователя и зону ответственности;
- дает понятные форматы выгрузки;
- заменяет разрозненные "экспорты ради экспорта" по разным экранам.

## 2. Ожидаемый результат (Definition of Done)
Считаем задачу выполненной, когда:
1. В системе есть единый центр отчетов (экран `Отчеты`) как главный вход для генерации.
2. Для ролей настроены допустимые типы отчетов и ограничения по данным.
3. Каждый отчет формируется по периоду + фильтрам ответственности.
4. Экспорт в `xlsx` работает стабильно и содержит служебный заголовок:
- тип отчета,
- период,
- кто сформировал,
- дата/время генерации.
5. На операционных экранах (`Суда`, `Ремонты`, `Задачи` и т.д.) убраны или понижены второстепенные кнопки "Экспорт", если они дублируют центр отчетов.
6. Есть журнал факта генерации отчета (минимум: кто, когда, какой тип, какие фильтры).
7. Пользователь без участия разработчика может получить нужный ему отчет по своей зоне ответственности.

## 3. Модель доступа по ролям
Ниже минимальный целевой набор для первой итерации.

1. Клиент:
- отчеты только по своим заявкам/ремонтам;
- без доступа к внутренним производственным метрикам других клиентов.

2. Рабочий:
- отчеты по своим задачам/трудозатратам/выполнению.

3. Мастер:
- отчеты по своей бригаде (подчиненные рабочие);
- сводка по задачам и статусам выполнения в зоне мастера.

4. Оператор:
- отчеты по своему доку;
- сводка ремонтов, заявок, простоев, производительности.

5. Диспетчер:
- отчеты по всем операторам в зоне диспетчера;
- междоковая сводка.

6. Админ:
- эксплуатационные отчеты (аудит, доступы, качество данных, активность пользователей);
- не операционные решения "вместо" диспетчера/оператора.

## 4. Поэтапный план

## P1. Нормализация и единая точка входа
1. Зафиксировать в UI, что "Отчеты" — основной модуль генерации.
2. Провести ревизию всех кнопок "Экспорт" в других разделах:
- оставить только контекстный быстрый экспорт там, где он реально нужен;
- убрать дубли, которые конфликтуют с центром отчетов.
3. Унифицировать фильтры на странице отчетов:
- период,
- тип отчета,
- зона ответственности (зависит от роли).
4. Вынести единый контракт запроса на отчет во фронтенде (`services/reports.ts` + типы).

Результат P1:
- пользователь понимает, где строятся отчеты;
- интерфейс не создает ложных "экспортов".

## P2. Генерация реальных файлов и аудит
1. Реализовать backend endpoint(ы) генерации отчетов по единому DTO:
- role-aware проверки доступа;
- фильтрация по зоне ответственности.
2. Реализовать генерацию `xlsx` (минимум 1-2 ключевых отчета):
- "Отчет по ремонтам" для оператора;
- "Отчет по задачам бригады" для мастера.
3. Добавить аудит генерации отчетов:
- userId/email,
- reportType,
- период,
- фильтры,
- timestamp.
4. Во фронтенде: корректная обработка статусов генерации/ошибок.

Результат P2:
- отчеты не имитируются, а формируются из БД;
- есть трассировка, кто что генерировал.

## P3. Расширение библиотеки отчетов
1. Добавить остальные ролевые отчеты (клиент, диспетчер, админ).
2. Добавить преднастроенные шаблоны фильтров для частых сценариев.
3. При необходимости добавить `csv/pdf` после стабилизации `xlsx`.
4. Добавить smoke/e2e проверки ключевых сценариев генерации.

Результат P3:
- отчеты покрывают все роли;
- модуль масштабируется без хаоса в UI.

## 5. Технический контур изменений
Фронтенд (`frontend/src`):
1. `pages/Reports.tsx` — основной UX сценарий генерации.
2. `services/reports.ts` — единая интеграция с backend API.
3. `types/*` — типизация request/response под отчеты.
4. `utils/export.ts` — оставить только клиентские helper-операции, не подменяя backend-генерацию.
5. Ревизия кнопок экспорта в:
- `pages/Repairs.tsx`
- `pages/Ships.tsx`
- `pages/Tasks.tsx`
- и других местах, где есть дубли.

Бэкенд (`src/main/java`):
1. Endpoint(ы) отчетов с проверкой роли и зоны ответственности.
2. Сервис сборки данных (репозитории + маппинг в табличную модель).
3. Генератор xlsx.
4. Аудит генерации.

## 6. Критерии приемки по этапам

P1:
1. Один очевидный вход в отчеты.
2. Нет дублирующих/псевдо-экспортов, вводящих в заблуждение.

P2:
1. Минимум два отчета генерируются из реальных данных.
2. Есть проверка прав и ограничение зоны ответственности.
3. Появился аудит генерации.

P3:
1. Все роли покрыты отчетами по своей зоне.
2. Экспорт стабилен, понятен и проверяем.

## 7. Что делаем следующим шагом
Практический старт реализации:
1. Ревизия текущего `Reports.tsx` и `services/reports.ts`.
2. Инвентаризация всех кнопок "Экспорт" на страницах.
3. Внедрение P1 в UI с минимальными и прозрачными правками.
