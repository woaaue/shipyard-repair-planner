# Единый документ процессов по ролям (v1)

## Цель
Зафиксировать:
- текущие процессы ролей (as-is);
- целевые процессы (to-be);
- пробелы между ними;
- приоритет внедрения.

Документ синхронизирован с текущей моделью backend + новыми возможностями:
- `User.reportsTo` и `GET /api/users/{id}/subordinates`;
- `WorkItem.assignee` и `WorkItem.reviewStatus`;
- `Repair.operator` и операции назначения оператора.

## Роли и зона ответственности

| Роль | Основная ответственность | Вход | Выход |
|---|---|---|---|
| CLIENT | Формирование запроса и приемка результата | Заявка/ремонт/уведомления | Подтверждение ожиданий и приемки |
| WORKER | Выполнение назначенных работ | Назначенный `WorkItem` | Выполненная работа на проверку |
| MASTER | Назначение и приемка работ бригады | Рабочие, `WorkItem`, `Issue`, `Downtime` | Принятые работы и эскалации |
| OPERATOR | Оперативное управление ремонтом в доке | Ремонты оператора, мастера, исключения | Стабильный ход ремонта и эскалации диспетчеру |
| DISPATCHER | Распределение мощностей между операторами/доками | Загрузка доков, операторы, ремонты | Назначения и приоритеты верхнего уровня |
| ADMIN | Целостность доступа и оргструктуры | Пользователи, роли, иерархия | Корректная модель прав и подчиненности |

## CLIENT

### As-is
1. Создает `RepairRequest`.
2. Отслеживает статусы заявки и ремонта.
3. Видит `WorkItem` и уведомления.
4. Не работает с внутренними `Issue`.

### To-be
1. Полный клиентский цикл: заявка -> ремонт -> приемка.
2. Прозрачный прогресс по работам и изменениям.
3. Явная фиксация приемки результата.

### Пробелы
1. Нет явного endpoint процесса приемки.
2. Ограниченная история событий для клиента.
3. Нет клиентских вложений как модели.

### Приоритет
`Medium` (после стабилизации внутренних процессов производства).

## WORKER

### As-is
1. Получает список работ.
2. Меняет статус выполнения.
3. Ведет `notes`.

### To-be
1. Видит только свои задачи через `assigneeId`.
2. Переводит выполненное в `PENDING_REVIEW`.
3. Получает обратную связь мастера по проверке.

### Пробелы
1. Нужен четкий UX очередей: в работе, на проверке, возвращено.
2. Нужна причина возврата на доработку.
3. Нужен видимый таймлайн изменений задачи.

### Приоритет
`High` (базовый производственный контур).

## MASTER

### As-is
1. Управляет `WorkItem` по ремонту.
2. Фиксирует `Issue` и `Downtime`.
3. Контролирует выполнение бригады.

### To-be
1. Видит подчиненных через `subordinates`.
2. Назначает задачи рабочим.
3. Принимает или возвращает результат проверки (`reviewStatus`).
4. Эскалирует исключения оператору.

### Пробелы
1. Слабое разделение потоков назначения и приемки.
2. Нет обязательной причины отклонения.
3. Нет экрана загрузки бригады.

### Приоритет
`High` (ключевой управляющий уровень цеха).

## OPERATOR

### As-is
1. Ведет ремонты в доке.
2. Контролирует прогресс и исключения.
3. Координирует мастеров.

### To-be
1. Работает по ремонтам через `operatorId`.
2. Управляет мастерами в подчинении.
3. Ведет межремонтные приоритеты.
4. Эскалирует диспетчеру системные риски.

### Пробелы
1. Нет формализованного флага риска по ремонту.
2. Нет операционного журнала решений.
3. Слабая типизация эскалации.

### Приоритет
`High` (оперативная устойчивость ремонта).

## DISPATCHER

### As-is
1. Видит загрузку доков и пул ремонтов.
2. Координирует операторов.
3. Принимает верхнеуровневые решения.

### To-be
1. Назначает оператора ремонту.
2. Балансирует доки и операторов.
3. Ведет очередь критичных решений.
4. Контролирует эффект от переназначений.

### Пробелы
1. Нет единой очереди диспетчерских решений.
2. Нет прозрачных критериев эскалации.
3. Недостаточно прогнозных индикаторов по загрузке.

### Приоритет
`High` (планирование и перераспределение мощностей).

## ADMIN

### As-is
1. Управляет пользователями и ролями.
2. Настраивает подчиненность через `reportsTo`.
3. Поддерживает доступ и аудит.

### To-be
1. Валидирует оргструктуру на уровне правил.
2. Безопасно проводит смены ролей и подчиненности.
3. Обслуживает целостность модели (без сирот и циклов).

### Пробелы
1. Нет продвинутого валидатора структуры.
2. Нет мастера массовых переназначений.
3. Ограниченный UX расследований в аудите.

### Приоритет
`Medium` (критично для качества администрирования, но не блокирует ежедневную операционку).

## Сквозные процессы между ролями

## 1) Основной производственный поток
1. CLIENT создает заявку.
2. DISPATCHER планирует и назначает оператора на ремонт.
3. OPERATOR координирует мастеров по ремонту.
4. MASTER назначает работы рабочим.
5. WORKER выполняет и отправляет на проверку.
6. MASTER принимает или возвращает на доработку.
7. OPERATOR закрывает операционные риски и ведет ремонт до завершения.
8. CLIENT получает завершенный результат и подтверждает приемку.

## 2) Поток исключений
1. WORKER или MASTER фиксирует проблему/простой.
2. MASTER проводит первичную оценку и влияние.
3. OPERATOR принимает локальное решение или эскалирует.
4. DISPATCHER принимает междоковое решение при системном влиянии.
5. CLIENT получает корректные уведомления по срокам/статусу.

## Приоритеты внедрения (по этапам)

| Этап | Фокус | Роли | Приоритет |
|---|---|---|---|
| 1 | Персональные назначения и проверка работ | WORKER, MASTER | P0 |
| 2 | Управление ремонтами по оператору | OPERATOR | P0 |
| 3 | Распределение операторов и контроль загрузки | DISPATCHER | P0 |
| 4 | Стабилизация иерархии и админ-процедур | ADMIN | P1 |
| 5 | Клиентская приемка и расширенная прозрачность | CLIENT | P1 |

## Что уже поддержано backend
1. Иерархия подчиненности (`reportsTo`, `subordinates`).
2. Назначение исполнителя задачи (`assignee`) и проверка (`reviewStatus`).
3. Назначение оператора ремонта (`operator`).

## Что требует доработок backend
1. Формализованная клиентская приемка как отдельная операция.
2. Структурированные причины возврата задачи на доработку.
3. Унифицированный журнал событий по сущностям для всех ролей.
4. Расширенная модель вложений/документов.
